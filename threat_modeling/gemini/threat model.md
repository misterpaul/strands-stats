 Strands Stats - Threat Model

**Owner:** Paul Faulstich
**Date:** 2025-08-15

## 1. Overview

This document outlines the potential security threats to the Strands Stats application. The application's primary function is to receive game results via email, process them through a serverless pipeline, and store the structured data in a Firestore database for later viewing in a web application.

## 2. Architecture Diagram

The following diagram illustrates the major components and data flows within the system.

![Architecture Diagram](./architecture_diagram.svg)

**Data Flow:**
1.  A **Game Player** sends an email containing game results to a dedicated Gmail account.
2.  The **Gmail** service receives the email. A filter applies a specific label.
3.  An **Apps Script** trigger runs periodically, reads labeled emails, validates the sender, and uploads the raw email content as a `.eml` file to a GCS bucket.
4.  The file creation in the **GCS Bucket** generates an Eventarc event, which triggers the **Cloud Function**.
5.  The **Cloud Function** reads the `.eml` file, parses it, and writes the structured game data to the **Firestore DB**.

|:--- |:--------------------------------------------------------------------------------------------------------------------------------------------------------|:--------------------------------- |:-----------------------|:------|:------------------------------------------------------------------------------------------------- |
| T01 | **Cost-based Denial of Service via Email Flooding**                                                                                                     | Game Player, Apps Script, Cloud Function            | Denial of Service      | High   | **Open.** The `ALLOWED_SENDERS` list is a partial mitigation. A stronger mitigation is to implement rate-limiting in the Apps Script (e.g., check a Firestore collection for the last submission time for that user and only allow one email per user per day). GCP Billing Budgets and alerts should be configured in OpenTofu as a critical guardrail. |
| T02 | **Unauthorized Data Access via Weak Firestore Rules**                                                                                                   | Firestore DB, Web App (Future)                      | Information Disclosure | High   | **Open.** Implement and enforce Firestore Security Rules that restrict data access based on the authenticated user's UID. For example: `allow read: if request.auth.uid == resource.data.userId;`. These rules must be developed and deployed via the Firebase CLI before the web app goes live. |
| T03 | **Malformed Email Causes Repeated Function Failures**                                                                                                   | Cloud Function                                      | Denial of Service      | Medium | **Mitigated.** The Cloud Function's main logic is wrapped in a `try...catch` block. On failure, the `catch` block calls the `archiveFile` function to move the problematic file to a dedicated `failures` bucket. This isolates the "poison pill" and allows other messages to be processed. |
| T04 | **Email Sender Spoofing**                                                                                                                               | Game Player, Apps Script                            | Spoofing               | Medium | **Open.** This is partially mitigated by modern email standards like SPF, DKIM, and DMARC, which Gmail uses to identify potential spoofing. The application implicitly relies on Gmail's filtering. For a low-risk project, this may be an acceptable risk. |
| T05 | **Insecure GCS Bucket Permissions**                                                                                                                     | GCS Bucket                                          | Information Disclosure | Low    | **Mitigated.** This is well-mitigated by the project's use of Infrastructure as Code. The OpenTofu configuration explicitly sets `public_access_prevention = "enforced"` on all buckets. Managing infrastructure via code significantly reduces the risk of manual error and configuration drift. |
| T06 | **Elevation of Privilege via Cloud Function Dependency**                                                                                                | Cloud Function                                      | Elevation of Privilege | Low    | **Open.** The risk is lowered by using a dedicated, least-privilege service account for the function. Regularly update dependencies (`npm audit`, Dependabot) to patch known vulnerabilities. The parsing logic should be treated as untrusted input and be as simple as possible. |
